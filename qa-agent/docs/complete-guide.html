The cast

Seven core agents + a validator + an optional AIO sync

You only type /qa-run <KEY>. The QA AZM Digital Agent dispatches these agents in order and runs the validator after every stage. When AIO Tests is enabled, the optional qa-test-sync also mirrors the approved plan into your test-management tool. Hover a card; explore the live workflow in the next section.

1 · qa-story

Reads the Jira story, normalizes its acceptance criteria into atomic, testable items.

2 · qa-test-writer

Turns every AC into concrete cases — happy path, negative, edge — with steps & expected results.

3 · qa-gap-analyzer

Proves every AC is exercised by ≥1 real test; requests more if a gap remains.

4 · qa-test-executor

Drives your live app in a real browser (Playwright): runs steps, captures screenshots & console errors.

5 · qa-bug-logger

Drafts fully-detailed bugs (Phase A); creates only the ones you approve, linked in Jira (Phase B).

6 · qa-reviewer

Computes coverage and a GO / NO-GO verdict from the real results.

★ qa-validator

After each stage, independently re-checks the work against the source and loops it back on any gap.

＋ qa-test-sync · optional

When AIO is enabled: creates the approved cases in the story's AIO Tests folder (named with the story key) and links each to the Jira story.

Test-management integration

AIO Tests sync — and the one folder you must create 📁

When enabled, qa-test-sync mirrors your approved test cases into AIO Tests for Jira (Cloud) so every user story gets a folder of detailed, linked test cases in the Cases module — each case carrying its steps, preconditions, the covered AC text, and a link back to the Jira story for traceability.

📁 Required: create the story folder first

AIO's public API cannot create folders (folder/set creation returns HTTP 500). So before the run you must create one folder in the AIO Cases module named with the user-story key — e.g. ABYR-2167 — using “Add new folder” under All. The agent then finds that folder by name and adds the story's test cases to it. Do this once per story.

What happens if the folder is missing

If no folder matching the story key exists, qa-test-sync does not create cases at the root — it stops and returns the exact name to create: No AIO folder found for <KEY>. Please create a folder named "<KEY>" under All in the AIO Cases module, then re-run. Create it, then re-dispatch the sync.

Enable it in three lines

  1. In .qa-config.json, set the aio block's enabled to true (and confirm projectKey).
  2. Add your AIO API token to the git-ignored .qa-secrets as AIO_TOKEN=….
  3. Create the story-key folder in the AIO Cases UI (above).
.qa-config.json · aio block
"aio": { "enabled": true, "baseUrl": "https://tcms.aiojiraapps.com/aio-tcms/api/v1", "tokenEnv": "AIO_TOKEN", "projectKey": "ABYR", "linkToStory": true }
On the ABYR-2167 run

With the ABYR-2167 folder pre-created, qa-test-sync created all 26 cases in it (ABYR-TC-…), each with tailored steps, precondition, the covered AC text, and linked to Jira story ABYR-2167 for traceability — written to aio-sync.json for the audit trail.

Run once per story

AIO's API can't delete cases either, so a re-run would create duplicates. qa-test-sync runs once per story and does not gate the browser execution — it's a side-integration on the way to running the tests.

Read this first

How credentials & the URL are handled 🔐

The agent logs into your app but is built so your password is never committed, printed, or shipped. Here's exactly where each thing lives.

ItemSaved whereIn the repo?
App URL (baseUrl, loginUrl).qa-config.json — plain text (not a secret)✗ git-ignored
Credential/token names (QA_USER/QA_PASS/AIO_TOKEN).qa-config.json — names only✗ git-ignored
Credential values + AIO token.qa-secrets — a git-ignored .env-style file, or OS env vars✗ never
Resolution order

Each secret is resolved by name: OS environment variable first → then the git-ignored .qa-secrets file → otherwise the step is blocked (never fabricated). The AIO token is read the same way, held in memory only, and never printed — all AIO calls send it as Authorization: AioAuth <token>.

Two more protections

Masking redacts emails, tokens, long digit sequences, and password/secret assignments before anything is written — values never reach results.json, screenshots, reports, Jira bugs, or AIO cases. And the executor auto-deletes the browser's .playwright-mcp/ snapshot scratch at the end of every run.

.qa-secrets · never committed
# .env-style — filled in privately, ignored by git QA_USER=tenantadmin001@test.local QA_PASS=•••••••••••• AIO_TOKEN=•••••••••••••••••••••••• # only needed if aio.enabled
Get it onto your machine

Download & install the agents — step by step

You install the agents once per machine; after that they work in every project via /qa-run and /qa-setup. Requires Windows + VS Code + Claude Code and PowerShell 5.1.

What you need first

  • VS Code + Claude Code active.
  • A staging URL for the app under test — a production-looking URL is refused by default (see Safety).
  • A test account for that app (for our run: tenantadmin001 on abyan-qc.azm.sa).
  • (Optional) AIO Tests — an AIO API token plus the story-key folder, only if you want the AIO sync (see the AIO Tests section above).

Step 1 · Get the agent code

Pick either option — both give you the qa-agent/ folder that holds all 8 agents + 2 commands:

Option A — download the release (no Git)
# 1. Open the repo's Releases page → latest release # 2. Under "Assets", click: Source code (zip) # 3. Right-click the downloaded .zip → Extract All (e.g. to C:\ManualTestingAgent)
Option B — clone with Git
git clone https://github.com/UsamaArshadJadoon/ManualTestingAgent.git
📦 The “Source code (zip)” is the agent code

That release asset contains the whole repo, including qa-agent\agents\ (the 8 qa-* agents), qa-agent\commands\ (qa-run, qa-setup), and the installer qa-agent\install.ps1. There's nothing else to download.

Step 2 · Authorize the connectors

In claude.ai → Settings → Connectors, authorize Atlassian (Jira) and Playwright. (Atlassian is also what qa-test-sync uses to link each AIO case back to the story.)

Step 3 · Install the agents

Open Windows PowerShell inside the extracted/cloned folder (the one that contains qa-agent\) and run the installer:

Windows PowerShell — install
powershell -File qa-agent\install.ps1 ✔ Installed QA AZM Digital Agent to ~/.claude agent: qa-story · qa-test-writer · qa-gap-analyzer · qa-test-executor agent: qa-bug-logger · qa-reviewer · qa-validator · qa-test-sync command: qa-run · qa-setup

Copies the 8 agents + 2 commands into ~/.claude, so they work in every project.

Step 4 · Verify the install

Windows PowerShell — verify
Get-ChildItem $HOME\.claude\agents\qa-*.md Get-ChildItem $HOME\.claude\commands\qa-*.md # expect 8 qa-* agents + qa-run.md and qa-setup.md
Updating later

To get a newer version, download/clone again and re-run powershell -File qa-agent\install.ps1 — it overwrites the deployed copies with the latest. Then continue to Step 1 of the walkthrough below (/qa-setup) in whatever project you want to test.

Why you can trust it

Safety & important points

  • Two human gates. Nothing runs against your app before the test-plan gate; nothing is written to Jira before the bug gate. You always control the consequential actions.
  • Production guard. A production-looking URL stops the run until you explicitly allow it. Keep allowProduction:false for safety.
  • Credential hygiene. Values (and the AIO token) live only in .qa-secrets/env, are masked everywhere, and browser snapshot scratch (.playwright-mcp/) is auto-deleted each run.
  • Self-correcting. The validator re-checks every stage from the source and loops back on gaps (max 2 retries) — how the 2 dropped AC were recovered in Step 4.
  • AIO sync is safe & optional. It runs only when enabled, needs a pre-created story folder, runs once per story (AIO can't delete), and never gates the browser run.
  • Non-destructive & flaky-aware. Prefers read/create over update/delete, reverts edits, retries a failing case before calling it a real bug; blocked ≠ failed.
  • Re-run & resume. --rerun re-tests only prior failures after a fix; --resume continues an interrupted run.
